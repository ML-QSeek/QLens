## 什么是 Q.lens？

Q.lens 是一套 **AI 原生应用开发范式**。

它不是框架，不是平台，不是编程语言。它是一套**设计理念、组织方式和交互模式**——让人类和 AI 用同一种方式理解系统、定义流程、追踪结果。

> **核心思想：以数据为中心，以函数为能力，以路径为流程。**

Q.lens 的目标是拆掉人类与 AI 之间的那堵墙。在传统架构里，AI 是“辅助写代码的工具”；在 Q.lens 里，AI 是“系统运行的参与者”。


## 核心理念

### 数据是活的
数据不再是“被处理的原材料”，而是“带着自己的修炼路线图进入系统的主角”。它流经各个原子块，被不断加工、校验、增强，每一步都有记录。

### 函数是能力
原子块是纯函数，有明确的输入输出签名、功能描述、版本号。可独立测试、独立演进、独立替换。

### 路径是流程
YAML 路径声明式描述数据修炼路线。**改路径即改流程，不需要改代码、不需要重新部署。**


## 核心组件

| 组件 | 职责 |
| :--- | :--- |
| **函数湖** | 所有原子块的集合，O(1) 哈希查找 |
| **原子块** | 纯函数，有签名、描述、版本 |
| **YAML 路径** | 声明式流程描述（串行/并行/分支） |
| **修炼日志** | 数据自带的完整修炼记录（审计追溯核心） |
| **数据管理器** | 管理修炼上下文，决定下一步去哪 |
| **函数管理器** | 原子块的注册、发现、版本管理 |


## 与传统架构的对比

| 维度 | 传统架构 | Q.lens 范式 |
| :--- | :--- | :--- |
| 组织方式 | 服务 / 类 / 模块 | 数据 + 函数 + 路径 |
| 流程定义 | 代码硬编码 | YAML 路径声明 |
| 系统理解 | 读代码推演调用链 | 读修炼日志 |
| AI 角色 | 辅助写代码 | 系统运行的参与者 |
| 变更方式 | 改代码 → 重新部署 | 改路径 → 热生效 |
| 审计追溯 | 多系统拼凑 | 数据自带修炼日志 |


## 核心工作流：数据修炼
数据进入 → 分配 cultivation_id

读取 YAML 路径 → 获取第一站

调用对应原子块（修炼站）

原子块执行 → 返回结果

更新修炼日志（记录这一站）

读取下一站 → 重复 3-5

所有站执行完毕 → 修炼完成，返回最终结果

## 设计原则

| 原则 | 说明 |
| :--- | :--- |
| **显式优于隐式** | 所有依赖都在 YAML 路径里显式声明 |
| **数据为主** | 数据带着自己的上下文，不依赖外部状态 |
| **AI 可读** | YAML + 修炼日志，结构化，AI 可解析 |
| **渐进演进** | 从单体到分布式平滑迁移，不推倒重来 |
| **轻量优先** | 不需要 K8s、不需要微服务，Redis + 进程即可启动 |
